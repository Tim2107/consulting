# Provider Routes And GDPR Posture

Status: board-facing draft for the AI Governance and Steering Board.

Date checked: 2026-05-29

This table summarizes common AI provider routes and their GDPR-relevant data handling posture. It is not a legal opinion and should not replace provider-specific legal, security, procurement, or client-contract review.

The key point for board decisions is that GDPR suitability is **route-specific**, not provider-brand-specific. The same model brand can have different data protection implications depending on whether it is used through a direct API, a cloud provider route, a workspace copilot, a free developer tier, a managed private deployment, or a self-hosted environment.

| Provider route | GDPR / DPA posture | Training policy | Storage / residency posture | Governance note |
| --- | --- | --- | --- | --- |
| OpenAI API | OpenAI's DPA says OpenAI acts as processor for customer data; EEA and Swiss customers contract with OpenAI Ireland. Sources: [OpenAI DPA](https://cdn.openai.com/pdf/openai-data-processing-addendum.pdf), [OpenAI API data controls](https://developers.openai.com/api/docs/guides/your-data). | API data is not used to train or improve OpenAI models unless the customer explicitly opts in. | Default abuse-monitoring logs may retain customer content for up to 30 days. Zero Data Retention (ZDR), Modified Abuse Monitoring (MAM), and data residency controls are available only for eligible customers, regions, models, and endpoints. | Candidate for restricted use only after DPA, retention, ZDR/MAM, endpoint, data residency, and feature review. |
| ChatGPT Enterprise / Business | Enterprise privacy commitments apply to business data in ChatGPT Business, Enterprise, Edu, Healthcare, and related enterprise workspaces. Source: [OpenAI Enterprise Privacy](https://openai.com/enterprise-privacy/). | OpenAI does not train on business data by default; opt-in feedback or sharing mechanisms can change that for shared data. | Retention, internal-source connections, GPTs, apps, and admin settings shape the actual data flow. | Good candidate for approved productivity use, but GPTs, apps, connectors, public sharing, retention settings, and employee transparency need review. |
| Azure OpenAI / Microsoft Foundry Models sold by Azure | Microsoft-operated route under Azure/Microsoft terms and data protection commitments. Source: [Microsoft Foundry data privacy](https://learn.microsoft.com/en-us/azure/foundry/responsible-ai/openai/data-privacy). | Prompts and completions are not stored in the model and are not used to train, retrain, or improve base models. | Prompts and responses are processed in the customer-specified geography unless using Global or DataZone deployment types. Global deployments may process in any geography where the model is deployed; DataZone deployments process within the specified data zone. Stored feature data remains in the customer-designated geography. | Preferred OpenAI-model route when the board requires Microsoft-operated processing. Global and DataZone deployments need explicit routing approval. |
| Microsoft 365 Copilot | Covered by Microsoft 365 commercial privacy, security, compliance, GDPR, and EU Data Boundary commitments, with model and feature caveats. Source: [Microsoft 365 Copilot privacy](https://learn.microsoft.com/en-ie/microsoft-365/copilot/microsoft-365-copilot-privacy). | Prompts, responses, and Microsoft Graph data are not used to train foundation LLMs. | Operates inside the Microsoft 365 service boundary and uses existing tenant permissions. Interaction data is stored. Some model choices, including Anthropic models in Copilot experiences, have EU Data Boundary caveats. | Treat as a Microsoft 365 tenant-permission and employee-transparency route, not as direct OpenAI. Requires permission hygiene, Purview/DLP review, connector and agent governance, audit review, and user notice. |
| Anthropic API / Claude for Work | For commercial products, Anthropic states that the customer is controller and Anthropic acts as processor. Source: [Anthropic processor/controller FAQ](https://support.claude.com/en/articles/9267385-does-anthropic-act-as-a-data-processor-or-controller). | Anthropic does not use commercial customer data to train models unless the customer chooses to participate in the Development Partner Program. | Anthropic commercial terms and DPA govern the direct route. Processing location, retention, support access, and enterprise controls must be checked for the specific plan and agreement. | Candidate route where Anthropic-operated processing is acceptable. Do not treat it as AWS-only or cloud-provider-operated simply because Claude is also available through cloud platforms. |
| Claude and other foundation models in Amazon Bedrock | AWS-operated route under AWS controls. AWS describes Bedrock as supporting GDPR and other compliance programs. Source: [Amazon Bedrock security and privacy](https://aws.amazon.com/bedrock/security-privacy-responsible-ai/). | AWS says Bedrock does not share inputs and outputs with model providers and does not use them to train base models. | Region and inference profile matter. In-region, geographic cross-region, and global routing have different processing-location implications. Customer-enabled invocation logs, knowledge bases, agents, and customization outputs must be governed separately. | Strong candidate where sensitive workloads require AWS-operated processing. Global or cross-region modes need explicit board approval where processing location matters. |
| Google Vertex AI / Gemini API on Google Cloud | Governed through Google Cloud terms and Cloud Data Processing Addendum. Sources: [Vertex AI and zero data retention](https://docs.cloud.google.com/vertex-ai/generative-ai/docs/vertex-ai-zero-data-retention), [Vertex AI data residency](https://docs.cloud.google.com/vertex-ai/generative-ai/docs/learn/data-residency). | Google says it will not use customer data to train or fine-tune AI/ML models without the customer's prior permission or instruction. | Data at rest stays in the customer-selected location. ML processing occurs in the region or multi-region where the request is made only for explicitly supported regional endpoints. Some features retain data, such as Search grounding, Maps grounding, or session resumption; in-memory caching has separate controls. | Candidate route for sensitive use only through supported regional or EU endpoints and with grounding, logging, caching, session, and feature controls reviewed. |
| Gemini for Google Workspace / Gemini Enterprise | Workspace customer data is governed by the Workspace agreement and Cloud Data Processing Addendum. Source: [Gemini Workspace Privacy Hub](https://knowledge.workspace.google.com/admin/gemini/generative-ai-in-google-workspace-privacy-hub). | Workspace does not use customer data to train generative AI models without the customer's prior permission or instruction. | Admins control conversation retention. Gemini uses existing Workspace permissions and may interact with Gmail, Drive, Docs, Sheets, Meet, Search grounding, NotebookLM, Chrome, and third-party apps depending on configuration. | Treat like Microsoft 365 Copilot: permission cleanup, DLP/data-region review, connector governance, employee transparency, and feature allowlists are central. |
| Google AI Studio / Gemini API free tier | Not the same posture as paid Google Cloud or Workspace routes. Source: [Gemini API pricing](https://ai.google.dev/gemini-api/docs/pricing). | Free-tier usage may be used to improve Google products. Paid-tier treatment differs. | Not suitable as an enterprise-controlled processing route for protected data. | Prohibit for Client Confidential Data, employee-sensitive data, secrets, contracts, source code requiring protection, and regulated financial-sector material. |
| Mistral API / Le Chat Enterprise | Mistral provides GDPR-rights materials, commercial terms, and a Data Processing Addendum. Sources: [Mistral privacy docs](https://docs.mistral.ai/admin/security-access/privacy), [Mistral commercial terms](https://legal.mistral.ai/terms/commercial-terms-of-service). | API data is not used for model training. Le Chat Pro, Team, and Enterprise conversations are not used for model training by default, with plan and opt-out caveats. | Product route, plan, Labs models, connectors, libraries, hosting location, retention, and support access must be reviewed. | Attractive EU provider option, but still requires route-specific DPA, hosting, retention, subprocessor, transfer, connector, and plan review. |
| xAI API / Grok Business or Enterprise | xAI says GDPR and SOC information is available through its Trust Center under NDA; enterprise terms are required for protected data. Sources: [xAI API security](https://docs.x.ai/developers/faq/security), [xAI regional endpoints](https://docs.x.ai/developers/regions). | API inputs and outputs are not used for training without explicit permission. | Default API requests and responses are stored for 30 days for abuse or misuse auditing. Enterprise ZDR can prevent request/response storage. Default endpoint auto-routes for latency; regional endpoints can constrain processing, and data-at-rest residency requires sales agreement. | Candidate only with enterprise contract, DPA/transfer review, ZDR confirmation, regional endpoint enforcement, audit-log review, and data-at-rest terms. Consumer/personal Grok remains prohibited for protected data. |
| Consumer or personal AI apps | Usually not an enterprise DPA-controlled route. Terms, training use, retention, human review, support access, and regional controls vary by product and account type. | Policies may allow product improvement, model training, feedback review, or human review depending on product and settings. | Retention, deletion, export, audit, residency, and subprocessor controls are typically weak or not suitable for business-critical protected data. | Prohibit for Client Confidential Data, secrets, credentials, client contracts, regulated data, protected source code, and delivery-pipeline material. |
| Company-operated self-hosted or managed private open-weight route | GDPR responsibility remains with the company or client as controller/operator. A managed private deployment still requires provider DPA and subprocessor review. | No external provider training unless the company deliberately trains or fine-tunes. | Data can stay inside the approved boundary only if inference, embeddings, indexes, logs, backups, monitoring, telemetry, and support access are also inside or contractually constrained to that boundary. | Best fit where Client Confidential Data must not leave company or client-controlled infrastructure. Creates operational ownership cost and still needs security, access, deletion, audit, and license review. |

## Board-Level Reading

No route should be approved only because the provider says "no training." For GDPR and Client Confidential Data workflows, the board should also require:

- DPA or processor terms, including subprocessors and international-transfer mechanism.
- Processing location for inference, storage, logs, backups, support access, and telemetry.
- Retention controls for prompts, outputs, files, embeddings, vector stores, traces, conversation history, abuse monitoring, and application state.
- ZDR, MAM, regional endpoint, DataZone, EU endpoint, or private-deployment controls where needed, with technical enforcement.
- Review of connectors, web search, grounding, agents, code execution, browser access, remote tools, and file uploads.
- Employee AI Transparency Notice when prompts, files, meetings, telemetry, or usage logs can be tied to employees.
- Client Contract Compatibility Check before any Client Confidential Data leaves the Company-Controlled Environment.
- Separate board/legal/client approval before training or fine-tuning on company or client material.
